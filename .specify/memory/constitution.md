# 餐厅项目宪章

## 核心原则

### I. 代码质量不可妥协
所有生产代码必须清晰、内聚，并且范围明确。模块必须有清楚的职责边界、
命名和所有权，避免隐藏耦合。重复逻辑只在造成维护风险时抽取；新增抽象
必须服务于当前复杂度，而不是基于猜测的未来需求。不得合入死代码、
注释掉的实现代码，或没有解释的 TODO。

### II. 可验证交付
每个用户可见行为、校验规则和业务流程，在完成前都必须有可重复的验证方式。
当项目技术栈支持自动化测试时，新功能必须包含自动化测试；如果暂时无法
自动化，任务中必须写明手动验证步骤和预期结果。回归修复必须包含一个会在
修复前失败的测试，或可复现的问题说明。

### III. 用户体验一致性
应用必须在客户侧和管理侧流程中保持一致的交互模型、视觉层级、文案风格和
响应式行为。主要操作必须清晰可见，表单必须提供明确的校验反馈，页面必须
在移动端和桌面端都保持可用。语义结构、键盘可达性、足够的对比度等基础
可访问性要求，必须作为功能定义的一部分。

### IV. 性能纪律
功能必须定义与用户旅程相匹配的性能预期。交互页面应在典型消费设备上保持
足够快的首屏加载和常用操作响应。昂贵的渲染、网络、存储或计算路径，必须
在计划中识别，并在完成前验证。性能优化必须基于可测量瓶颈，或边界清晰的
风险判断。

### V. 简单性与可维护性
项目必须优先采用能满足当前规格、同时便于未来修改的最简单设计。技术选择
必须在计划中记录取舍。共享状态、持久化和跨模块契约必须通过清晰接口隔离，
使 mock 或本地实现未来可以替换为真实后端，而不需要重写用户流程。

## 产品约束

餐厅应用必须优先服务真实的顾客和员工工作流，而不是只做装饰性或营销型页面。
菜单浏览、预订请求、点单/购物车请求、联系信息和后台审核流程，必须能够被
独立理解和测试。面向顾客的表单必须说明提交结果是“请求”还是“已确认动作”。
在实现认证和授权之前，后台功能必须明确标注为开发阶段占位能力。

## 开发流程

工作必须按照 Spec Kit 流程推进：宪章、规格、计划、任务、实现。规格必须先
描述用户需要什么以及为什么需要，再决定技术选型。计划必须包含宪章检查，
覆盖代码质量、测试、用户体验一致性、可访问性、性能和可维护性。任务必须
足够小、可独立验证，并尽量按用户故事分组。完成工作前必须运行可用的 lint、
typecheck、test 和 build 命令；如果某项检查不可用，必须说明原因。

## 治理

本宪章优先于本项目中与其冲突的局部实践。任何修订都必须通过更新本文档完成，
并包含同步影响报告，说明变更的原则、受影响模板和后续工作。版本号遵循语义化
版本：MAJOR 用于不兼容的治理变更或原则移除，MINOR 用于新增原则或实质扩展要求，
PATCH 用于不改变语义的澄清。每个计划和评审都必须在实现开始前、以及工作完成前
验证其符合当前宪章。
