# UI 与工作流契约：餐厅 Web 应用

## 导航契约

- 顾客必须能从主导航进入：首页、菜单、预订、点单/购物车、联系信息。
- 后台入口必须可发现，但在未实现认证前必须显示开发占位提示。
- 移动端导航必须可键盘访问，且不得遮挡主要内容。

## 首页契约

- 首页必须展示餐厅名称、餐饮定位、营业时间、地址、联系方式和主要行动入口。
- 主要行动入口至少包含：查看菜单、预订、开始点单。
- 首页不得只是营销说明，必须能进入实际工作流。

## 菜单契约

- 菜单必须按分类展示。
- 菜品卡片必须展示名称、描述、价格、标签和可售状态。
- 饮食标签筛选后，结果必须立即可见。
- 筛选无结果时必须提供空状态和清除筛选方式。
- 不可售菜品必须显示状态，并禁止加入购物车。

## 预订契约

- 预订表单必须包含：姓名、联系方式、日期、时间、人数、备注。
- 缺失必填字段、过去日期、无效时间、无效人数必须阻止提交。
- 提交成功后必须说明“这是预订请求，餐厅会确认 availability/可用性”。
- 新预订状态必须为待确认。

## 点单/购物车契约

- 顾客必须能添加可售菜品、调整数量、删除购物车项。
- 购物车必须展示每项数量、小计和订单总计。
- 空购物车不得提交。
- 缺失顾客姓名、联系方式或取餐方式不得提交。
- 提交成功后必须说明“这是点单请求，餐厅会确认订单”。
- 新订单状态必须为待确认。

## 后台契约

- 后台必须展示待处理预订数量、待处理订单数量和不可售菜品数量。
- 菜单管理必须支持新增、编辑、删除或下架菜品。
- 预订管理必须支持查看请求并更新状态。
- 订单管理必须支持查看明细并更新状态。
- 未实现认证前，后台页面必须显示不可用于生产环境的提示。

## 响应式与可访问性契约

- 移动端和桌面端均不得出现主要内容遮挡、按钮文字溢出或不可点击状态。
- 主要表单字段必须有可读标签和靠近字段的错误反馈。
- 主要操作必须可通过键盘访问。
- 状态变化必须通过文字或明确视觉标签表达，不能只依赖颜色。
