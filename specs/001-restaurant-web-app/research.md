# 研究记录：餐厅 Web 应用

## 决策：采用前端单页应用作为第一版交付形态

**理由**：当前项目为空，需求集中在顾客流程、菜单交互、表单提交、购物车和后台演示管理。前端单页应用可以最快完成可体验 MVP，并降低后端和部署复杂度。

**考虑过的替代方案**：
- 全栈应用：更接近生产形态，但会在第一版引入数据库、接口、认证和部署决策。
- 纯静态页面：实现更快，但无法自然支持购物车、预订请求、订单状态和后台管理。

## 决策：第一版使用本地/mock 数据访问层

**理由**：规格没有要求真实后端或多用户同步。使用 repository 层封装数据读写，可以先完成完整流程，同时为未来切换后端保留接口边界。

**考虑过的替代方案**：
- 直接在组件中写死数据：简单但难以维护，后台修改和顾客侧同步会变复杂。
- 立即接入数据库：过早增加基础设施负担，不符合 MVP 范围。

## 决策：预订和点单均作为“请求”处理

**理由**：餐厅通常需要确认桌位、库存、人力和营业状态。规格也明确第一版不包含自动桌位库存、支付或配送承诺。

**考虑过的替代方案**：
- 即时确认：用户反馈更强，但暗含库存、支付、桌位和履约保证。
- 只收集联系表单：更简单，但无法覆盖预订和点单的核心业务流程。

## 决策：后台作为基础管理与演示能力

**理由**：后台需要支撑菜单、预订和订单状态管理，但没有认证授权前不能宣称生产可用。显式标注占位能力符合宪章。

**考虑过的替代方案**：
- 不做后台：顾客侧可以演示，但菜单/状态管理闭环不完整。
- 做完整权限系统：生产价值更高，但超出第一版范围。

## 决策：验证采用自动化优先、手动验证兜底

**理由**：项目尚未建立测试栈。任务阶段应优先创建可运行的 lint/typecheck/build 和关键交互测试；若某项自动化暂不可用，需要在 quickstart 中保留明确手动验证步骤。

**考虑过的替代方案**：
- 只手动验证：短期更快，但回归风险高。
- 先搭完整测试体系：质量更强，但可能压过 MVP 实现本身。
